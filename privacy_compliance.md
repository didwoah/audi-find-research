# Audi-Find: 개인정보 보호법 준수 검토

> audi_find_encryption_design.md의 설계를 기준으로, GDPR(EU), PIPA(한국), CCPA(캘리포니아) 적용 여부와
> 각 법적 요건 충족 여부를 검토한다.

---

## 1. 핵심 전제: Mel-Spectrogram은 개인정보인가?

가장 먼저 짚어야 할 질문이다. "Raw 오디오가 아니라 주파수 변환 벡터를 전송하므로 개인정보가 아니다"라는 주장은 **법적으로 성립하지 않는다.**

### GDPR Article 4(1) — 개인정보의 정의

> "identified or identifiable natural person에 관한 모든 정보"

Mel-Spectrogram이 개인정보에 해당하는 이유는 다음과 같다.

```
[직접 식별 가능성]
  Mel-Spectrogram에는 화자(話者)의 성문(voiceprint) 특성이 남아 있음.
  주변에 사람이 있었다면 화자 식별 모델로 역추적 가능.
  → '이 사람이 이 장소에 있었음'을 식별 가능한 정보.

[간접 식별 가능성]
  Index_i(암호화 인덱스) + 수집 타임스탬프 + Finder GPS 위치
  → 특정 시간·장소에 버즈 소유자가 있었음을 추론 가능.
  → GDPR Recital 26: "모든 합리적 수단을 사용하여 식별 가능한 경우" 개인정보 해당.

[특수 범주 데이터 위험]
  주변 음향에는 다음이 포함될 수 있음:
    - 종교 집회 소리 (교회, 사원, 모스크)    → Article 9 특수 범주
    - 의료 기관 특유의 소음 (병원 방송, 의료 기기)  → 건강 정보 추론 가능
    - 집회·시위 현장 소음                         → 정치적 견해 추론 가능
    - 타인의 대화 내용
```

**결론**: Mel-Spectrogram은 GDPR 상 개인정보(personal data)에 해당한다. 처리 행위에 따라 특수 범주 데이터(Article 9)로 분류될 가능성도 있다.

## 2. 제3자 오디오 문제 — 가장 심각한 법적 리스크

Audi-Find의 설계상 **버즈 소유자 외의 타인**이 수집된 오디오에 등장한다.

```
시나리오:
  카페에서 버즈를 분실 → 주변 대화 소리가 Mel-Spectrogram에 포함
  지하철에서 분실     → 타인의 목소리, 통화 내용 포함 가능

법적 문제:
  해당 타인들은:
    (a) 자신의 음성이 수집되는지 인지하지 못함
    (b) 동의를 제공하지 않음
    (c) 자신에 관한 정보가 처리된다는 사실조차 모름
```

이는 GDPR Article 13/14(정보 제공 의무)와 Article 6(적법한 처리 근거) 위반 가능성이 있다.

**이 문제는 암호화 설계로 해결할 수 없다.** 암호화는 저장·전송 보안의 문제이고, 제3자 동의는 데이터 수집 자체의 적법성 문제이기 때문이다.

## 3. GDPR 적합성 검토 (EU·영국)

### 3.1 Article 6 — 처리의 적법한 근거

GDPR은 6가지 처리 근거 중 하나가 있어야 한다.

| 근거 | 내용 | Audi-Find 적용 가능성 |
|---|---|---|
| (a) 동의 | 정보 주체의 명시적 동의 | ✅ 가장 현실적. 단, 제3자 동의 불가 |
| (b) 계약 이행 | 계약 이행에 필수적인 경우 | △ "위치 찾기" 서비스의 핵심 기능으로 볼 수 있으나 다툼 여지 있음 |
| (c) 법적 의무 | 법적 의무 준수 | ❌ 해당 없음 |
| (d) 생명 보호 | 중요한 이익 보호 | ❌ 해당 없음 |
| (e) 공익 | 공공 과업 수행 | ❌ 해당 없음 |
| (f) 정당한 이익 | 컨트롤러의 정당한 이익 | △ 강력한 보안 설계로 주장 가능하나, 제3자 음성 포함 시 이익 균형 테스트 통과 불투명 |

**권고**: 처리 근거 **(a) 명시적 동의**를 주된 근거로 설정. 동의 없이 오디오 수집이 시작되면 안 됨. Find 기능 활성화 시 별도 동의 플로우 필수.

---

### 3.2 Article 5 — 개인정보 처리 원칙 적합성

| 원칙 | 요건 | 현재 설계 평가 |
|---|---|---|
| **적법성·공정성·투명성** | 정보 주체에게 처리 사실을 알려야 함 | ⚠️ 타인 오디오 포함 시 투명성 불가. 소유자에게는 명확한 고지 필요 |
| **목적 제한** | 수집 목적 외 사용 금지 | ✅ 위치 추론 단일 목적. LALM 학습 데이터 재사용 명시적 금지 필요 |
| **데이터 최소화** | 필요 최소한의 데이터만 수집 | ✅ Raw 오디오 → Mel-Spectrogram 변환으로 데이터량 감소. BLE 연결 중 "상시 수집" 설계는 재검토 필요 |
| **정확성** | 부정확한 데이터 삭제 또는 수정 | ✅ 추론 후 즉시 삭제 설계와 부합 |
| **보관 기간 제한** | 목적 달성 후 삭제 | ✅ 추론 완료 후 즉시 삭제 정책은 이 원칙에 부합. 단, "즉시 삭제"의 기술적 검증 수단 필요 |
| **무결성 및 기밀성** | 적절한 보안 조치 | ✅ ECIES + AES-256-GCM + TEE 설계는 Article 32 보안 요건을 충족하는 수준 |

---

### 3.3 Article 9 — 특수 범주 데이터

오디오에 건강 정보, 종교, 정치적 견해가 추론 가능한 형태로 포함된다면 Article 9 적용 대상이 된다.

```
Article 9(1): 특수 범주 데이터 처리는 원칙적으로 금지.
Article 9(2)(a): 명시적 동의 시 예외 허용.
```

**요건**: 일반 동의(Article 6)와 별도로 **특수 범주 데이터에 대한 명시적 동의**를 추가로 받아야 한다. 동의 문구에 "주변 소리에는 귀하 또는 제3자의 건강 상태, 종교, 정치적 활동 등을 추론할 수 있는 정보가 포함될 수 있습니다"와 같은 내용이 포함되어야 한다.

---

### 3.4 Article 25 — 개인정보 보호 중심 설계 (Privacy by Design)

GDPR이 요구하는 PbD(Privacy by Design) 원칙과 Audi-Find 설계의 대응 여부.

| PbD 원칙 | Audi-Find 설계 | 평가 |
|---|---|---|
| 사전 예방적 접근 | ECIES로 설계 단계에서 클라우드 접근 차단 | ✅ |
| 기본 설정으로서의 프라이버시 (Privacy as Default) | Find 기능 OFF가 기본값이어야 함 | ⚠️ 명시적 설계 필요 |
| 프라이버시 내재화 | Mel-Spectrogram 변환, 즉시 삭제 | ✅ |
| 전기능 (Positive-Sum) | 보안과 기능성 동시 달성 | ✅ |
| 종단간 보안 | 버즈에서 TEE까지 암호화 체인 | ✅ |
| 가시성 및 투명성 | 사용자 고지 및 감사 수단 | ⚠️ 미설계 |
| 이용자 중심 | 사용자가 수집 여부 통제 가능해야 함 | ⚠️ 미설계 |

---

### 3.5 Article 35 — 데이터 보호 영향 평가 (DPIA)

Article 35는 "혁신적 기술 사용", "대규모 특수 범주 데이터 처리", "민감한 지역 또는 위치 데이터와의 결합"에 해당하는 경우 **처리 시작 전 DPIA 수행을 의무화**한다.

```
Audi-Find의 DPIA 의무 해당 여부:

  ① 혁신적 기술 사용:
     LALM(대형 오디오 언어 모델) 기반 위치 추론 → 해당

  ② 민감 데이터 대규모 처리:
     음향 데이터(생체정보 추론 가능) → 해당

  ③ 위치 데이터와의 결합:
     GPS 후보 + 오디오 인덱스 → 해당

판정: DPIA 수행 의무 있음.
```

DPIA는 서비스 출시 전에 완료되어야 하며, 고위험이 잔존하는 경우 감독 기관(DPA)과 사전 협의 필요.

---

### 3.6 Article 17 — 삭제권 (Right to Erasure)

```
사용자 요청 시:
  클라우드에 저장된 [ Index_i | P_eph_i | C ] 전부 삭제 가능해야 함.

현재 설계 평가:
  Index_i는 SHA-256 해시 → 사용자가 A_i를 재계산하여 인덱스 목록 제시 가능.
  → 클라우드가 해당 인덱스 범위의 데이터를 삭제하는 API가 필요.

  주의: 클라우드가 Index_i와 실제 사용자를 연결하지 못하므로,
  사용자가 "내 데이터"를 특정하는 방법을 설계해야 함.
  → 스마트폰이 Index_i 목록을 클라우드에 제시하여 일괄 삭제 요청하는 방식이 적합.
```

---

### 3.7 Article 22 — 자동화된 의사결정

Qwen3.5-Omni-Plus의 위치 추론이 "법적 효과를 초래하거나 유사하게 중요한 영향을 미치는 자동화된 결정"에 해당하는지 검토.

```
현재 설계:
  LLM 추론 결과 = 지도 위 위치 표시 + 자연어 설명
  사용자가 최종 판단 → 단순 정보 제공

Article 22 적용 여부:
  법적 효과나 중요한 영향 없음 → 현재 설계에서는 Article 22 미적용.
  단, 추후 추론 결과가 보험, 법집행 등에 활용된다면 적용됨.
```

## 4. 한국 개인정보 보호법(PIPA) 적합성 검토

### 4.1 고유 식별 정보 / 민감 정보

```
개인정보보호법 제23조 (민감 정보):
  "건강에 관한 정보, 유전자 정보, 범죄 경력 정보,
   생체인식 정보, 신념·사상·정치적 견해 등"

음향 데이터와의 관련:
  성문(voiceprint) = 생체인식 정보 → 민감 정보 해당 가능
  병원 소음 포함 시 → 건강 정보 추론 → 민감 정보

처리 요건:
  제23조: 민감 정보는 별도 동의 + 처리 목적 명시 의무
  → GDPR Article 9와 유사.
```

### 4.2 개인정보 처리방침 의무

```
개인정보보호법 제30조:
  처리하는 개인정보 항목, 목적, 보유 기간, 제3자 제공 여부를
  공개 게시해야 함.

Audi-Find 적용:
  "Mel-Spectrogram(주변 음향 주파수 벡터)"을 처리 항목으로 명시.
  목적: "실내 위치 추론을 위한 단기 보관 후 즉시 삭제".
  보유 기간: "추론 완료 후 즉시 삭제 (최대 X초 이내)".
```

### 4.3 국외 이전

```
개인정보보호법 제28조의8 (2023년 개정):
  개인정보를 국외로 이전하려면:
    ① 정보 주체에게 이전 국가, 이전 항목, 이전 목적, 보호 조치 고지
    ② 정보 주체 동의 또는 표준계약조항(SCC) 등 안전장치 마련

Audi-Find 적용:
  클라우드 인프라(AWS, Azure, GCP)가 국외에 있을 경우 해당.
  국내 리전 우선 구성 또는 PIPA 제28조의8 요건 충족 필수.
```

## 5. CCPA (캘리포니아 소비자 프라이버시법) 적합성 검토

```
CCPA § 1798.140(o):
  "Audio, electronic, visual, thermal, olfactory, or similar information"
  → 오디오(Mel-Spectrogram 포함)는 CCPA의 개인정보에 명시적으로 포함됨.

CCPA 주요 요건 대비:

  ① 고지 의무 (§ 1798.100):
     수집 시점에 카테고리·목적 고지 필요.

  ② 삭제 요청권 (§ 1798.105):
     소비자 요청 시 보관 중인 데이터 삭제. Article 17 대응과 동일.

  ③ 판매 금지 옵트아웃 (§ 1798.120):
     Mel-Spectrogram을 제3자에게 판매하거나 공유하면 옵트아웃 권리 제공.
     Audi-Find는 LALM 학습에 데이터를 재사용하지 않음을 명시해야 함.

  ④ 민감 개인정보 (CPRA 추가, § 1798.121):
     "Precise geolocation" + 오디오 결합 = 민감 개인정보.
     → 사용 제한 옵트아웃 권리 제공 의무.
```

## 6. 설계 변경 없이 해결 불가능한 문제

### 문제 A: 제3자 음성 수집 (법적 해결 불가)

```
현실:
  Mel-Spectrogram에는 버즈 소유자 외 타인의 목소리가 포함됨.
  해당 타인은 동의를 제공하지 않음.

가능한 대응:
  ① VAD(Voice Activity Detection) 활성화 후 음성 구간 제거
     → BES2700YP에 하드웨어 VAD 탑재 (buds_hardware.md 확인)
     → 음성 제거 후 환경음(ambient sound)만 사용
     → 제3자 식별 가능성 대폭 감소

  ② 수집 시간 제한
     → BLE 연결 단절 직전 3초만 사용 (10초에서 축소)
     → 노출되는 오디오 범위 최소화

  ③ 위치 추론 후 Mel-Spectrogram 클라우드 보관 없이 실시간 처리
     → 저장 자체를 하지 않아 보관 리스크 제거
     → 단, BLE 단절 후 Finder 기기를 통해 전달하는 구조상 실시간 처리 불가
```

### VAD 처리 시간 분석

BES2700YP의 하드웨어 VAD와 HiFi4 DSP 소프트웨어 VAD의 처리 시간을 연산 과정과 함께 산출한다.

**기본 파라미터**

```
오디오: 10초, 16kHz, 16-bit mono
총 샘플 수: 10 × 16,000 = 160,000 샘플
VAD 프레임 크기: 20ms = 320 샘플  (VAD 업계 표준 단위)
총 프레임 수: 10,000ms / 20ms = 500 프레임
```

---

#### 방식 1: 하드웨어 VAD (BES2700YP 전용 회로)

BES2700YP에는 항상 켜져 있는(always-on) 전용 하드웨어 VAD 회로가 내장되어 있다.
동작 원리는 RMS 에너지 임계값 비교이며, 전용 아날로그-디지털 회로가 처리한다.

```
[하드웨어 VAD 동작 방식]

입력: 각 프레임의 RMS 에너지
  E_frame = sqrt( (1/N) × Σ x[n]² )   (N = 320 샘플)

전용 회로 동작:
  에너지 합산 → 하드웨어 누산기(accumulator)로 1 클럭 완료
  임계값 비교 → 비교기(comparator) 회로로 1 클럭 완료
  총 프레임당 처리: 2 클럭 사이클

BES2700YP always-on VAD 코어 클럭: ~1 MHz
프레임당 처리 시간: 2 / 1,000,000 = 2 μs
500 프레임 총 처리 시간: 500 × 2 μs = 1,000 μs = 1 ms
```

하드웨어 VAD는 **실시간(real-time)** 으로 스트리밍 오디오와 동시에 동작한다.
오디오 수집이 완료되는 시점에 VAD 레이블링도 이미 완료되어 있다.

```
[실시간 병렬 파이프라인]

t = 0 ms:    순환 버퍼 기록 시작 + 하드웨어 VAD 병렬 동작 시작
t = 20 ms:   프레임 1 오디오 수집 완료 → VAD 판정 즉시 완료 (2 μs, 비교 불가 수준)
t = 40 ms:   프레임 2 ...
...
t = 10,000 ms: 순환 버퍼 완성 + VAD 레이블 500개 완성

추가 처리 지연: 0 ms  ← 오디오 수집과 완전 병렬
```

---

#### 방식 2: HiFi4 DSP 소프트웨어 VAD (WebRTC VAD 알고리즘)

에너지 기반 하드웨어 VAD보다 정교한 GMM(Gaussian Mixture Model) 기반 소프트웨어 VAD를
HiFi4 DSP에서 오디오 수집 완료 후 후처리(post-processing)로 실행하는 경우다.

```
[WebRTC VAD 프레임당 연산량 추정]

1단계 — 주파수 대역 에너지 특징 추출:
  입력: 320 샘플 (20ms, 16kHz)
  연산: 6개 주파수 대역의 에너지 합산
    각 대역: 320 샘플 × 2 (곱셈 + 누산) = 640 MAC
    6개 대역: 640 × 6 = 3,840 MAC

2단계 — GMM 스코어링:
  구성: 2개 클래스(음성/비음성) × 6개 가우시안 컴포넌트 × 6개 특징 차원
  연산: 컴포넌트당 (평균 차이 × 분산 역수 × 지수함수) ≈ 10 연산
    2 × 6 × 6 × 10 = 720 연산
  로그합 계산 (log-sum-exp): 24 연산
    소계: ≈ 744 연산

3단계 — 사후확률·임계값 비교:
  ≈ 50 연산

프레임당 총 연산량: 3,840 + 744 + 50 = 4,634 연산  ≈ 5,000 연산 (반올림)
```

```
[HiFi4 DSP 처리 성능 계산]

BES2700YP HiFi4 DSP 클럭: 200 MHz
HiFi4 SIMD 폭: 64-bit → 1 사이클에 int16 MAC 4개 병렬 처리
유효 처리 속도: 200 MHz × 4 = 800 MMAC/s

프레임당 처리 사이클:
  5,000 연산 / 4 (SIMD) = 1,250 사이클

프레임당 처리 시간:
  1,250 / 200,000,000 = 6.25 μs

500 프레임 총 처리 시간:
  500 × 6.25 μs = 3,125 μs ≈ 3.1 ms
```

> 참고: WebRTC VAD 공식 문서는 10ms 프레임 @ 8kHz 기준 ~0.3% CPU (600MHz ARM Cortex-A)를 보고한다.
> 이를 20ms 프레임 @ 16kHz + HiFi4 SIMD로 환산하면 프레임당 ~6 μs 수준으로, 위 연산과 일치한다.

#### 음성 프레임 마스킹 처리 시간

VAD 판정 후 음성 프레임의 샘플을 0으로 덮어쓰는(masking) 과정의 시간을 산출한다.

```
[마스킹 연산량]

가정: 10초 오디오에서 평균 음성 구간 비율 30%
  (조용한 카페 실측 기준: 대화 있을 때 약 20~40%)

음성 프레임 수: 500 × 0.3 = 150 프레임
마스킹할 샘플 수: 150 × 320 = 48,000 샘플
쓰기 데이터량: 48,000 × 2 bytes = 96,000 bytes ≈ 96 KB

BES2700YP SRAM 대역폭:
  내부 버스 폭 32-bit, 클럭 200 MHz
  → 최대 대역폭: 200 MHz × 4 bytes = 800 MB/s

마스킹 처리 시간:
  96,000 bytes / 800,000,000 bytes/s = 0.12 ms
```

---

#### VAD 처리 시간 종합

| 방식 | VAD 연산 시간 | 마스킹 시간 | 합계 | 파이프라인 영향 |
|---|---|---|---|---|
| **하드웨어 VAD** | 0 ms (실시간 병렬) | 0.12 ms | **0.12 ms** | 사실상 무비용 |
| **HiFi4 소프트웨어 VAD** | 3.1 ms (후처리) | 0.12 ms | **3.2 ms** | 기존 15 ms에 3.2 ms 추가 |

```
비교 기준 (buds_hardware.md):
  Mel-Spectrogram 변환: ~10 ms
  AES-256-GCM 암호화:  ~2–5 ms
  기존 파이프라인 합계:  ~15 ms

VAD 추가 후:
  하드웨어 VAD 방식: 15 + 0.12 ≈ 15 ms  (변화 없음)
  소프트웨어 VAD 방식: 15 + 3.2 ≈ 18 ms  (+21%)

BLE 전송 시간 (~500 ms, ble_speed_optimization.md 기준)과 비교:
  하드웨어 VAD: 전체 지연의 0.02% 증가 → 무시 가능
  소프트웨어 VAD: 전체 지연의 0.6% 증가 → 무시 가능

결론: 두 방식 모두 BLE 전송이 병목인 현재 파이프라인에서 추가 지연 없이 도입 가능.
```

**권고 구현 전략**: 하드웨어 VAD(1차 에너지 필터, 0ms 추가 지연)로 명확한 음성 구간을 제거하고,
경계 판정이 불명확한 프레임에 한해 HiFi4 소프트웨어 VAD(2차 GMM 검증)를 적용하는
계층적(cascaded) 접근이 정확도와 전력 효율의 균형에 가장 적합하다.

### VAD 없이 제3자 문제를 피해가는 법적 논리

VAD 구현 없이 법적 논리만으로 제3자 음성 수집 문제를 방어하는 세 가지 접근을 검토한다.

---

#### 논리 1: Mel-Spectrogram은 제3자에 대한 개인정보가 아니다 (GDPR Recital 26)

GDPR Recital 26은 "합리적으로 가능한 수단을 동원해도 식별이 불가능한 경우"를 개인정보 범위에서 제외한다.

```
[역식별(re-identification) 난이도 분석]

제3자 식별을 위해 필요한 단계:
  ① Mel-Spectrogram 역변환 (inversion)
       Griffin-Lim, HiFi-GAN 등 vocoder 필요
       복원 음성 품질: MOS 2.5~3.0 수준 (원음 대비 열화)
       → 화자 인식 모델이 요구하는 MOS 4.0+ 에 크게 못 미침
  ② 화자 식별 모델(speaker verification) 실행
       d-vector, x-vector 등 별도 모델 필요
  ③ 비교 대상 voiceprint 데이터베이스 보유 필요
       → 일반적인 공격자가 보유하지 않음

결론:
  제3자 음성을 포함한 Mel-Spectrogram으로부터 특정 개인을 식별하는 데
  필요한 수단의 비용·시간·기술이 "합리적" 수준을 초과.
  → Recital 26 기준상 제3자에 대한 개인정보가 아닐 수 있음.
```

**논리 강화 조건**

이 논리가 성립하려면 설계 문서에 다음을 명시해야 한다.

```
① Qwen3.5-Omni-Plus 프롬프트에 화자 식별 태스크 없음
② 시스템 내 어디에서도 speaker verification 모델을 실행하지 않음
③ 처리 목적이 ASC(Acoustic Scene Classification) 단일 목적임을 처리방침에 기재
```

**약점**: GDPR 감독기관(특히 독일 DSK, 아일랜드 DPC)은 식별 가능성 판단에 보수적이다.
Mel-Spectrogram이 화자 인식에 활용 가능하다는 학술 연구가 존재하므로,
이 논리만으로 감독기관 조사를 완전히 방어하기 어려울 수 있다.

---

#### 논리 2: 정당한 이익 (Legitimate Interest, Article 6(1)(f))

제3자 음성에 대해 소유자 동의가 아닌 **Samsung의 정당한 이익**을 처리 근거로 사용하는 방법이다.
GDPR은 동의 외에도 컨트롤러의 정당한 이익이 정보 주체의 이익을 초과하는 경우 처리를 허용한다.

```
[3단계 균형 테스트 (Balancing Test)]

① 정당한 이익 존재 여부:
   - 분실 재산 회수 지원 (재산권 보호)
   - 서비스 기능상 필수적 요소
   → 인정 가능

② 처리의 필요성:
   - 음향 지문 없이는 실내 위치 판별 불가 (GPS 단독 100m 오차)
   - Mel-Spectrogram이 아닌 방법으로 동일 목적 달성 불가
   → 필요성 인정 가능

③ 제3자의 이익·권리와의 균형:
   + 제3자 식별 위험 낮음 (Mel-Spectrogram 역변환 품질 열화)
   + 처리 목적이 제3자와 무관 (위치 추론, 제3자 추적 아님)
   + 즉시 삭제로 보관 리스크 없음
   + 제3자가 입는 실질적 피해 거의 없음
   - 제3자가 처리 사실을 인지하지 못함 (투명성 부족)
   - 수집 규모가 Samsung의 사용자 수만큼 대규모
   → 균형 테스트 통과 가능성 있음 (단, 대규모 처리는 감점 요인)
```

**비교 사례**: CCTV는 거리를 지나는 불특정 다수를 동의 없이 촬영하지만,
EDPB WP 217 가이드라인에 따라 정당한 이익 근거로 합법 운영이 인정된다.
Audi-Find의 제3자 음향 수집은 CCTV의 영상 수집보다 식별 가능성이 낮으므로
동일하거나 더 유리한 판단을 받을 여지가 있다.

**약점**: 처리 대상이 특수 범주 데이터(건강·종교 추론 가능)를 포함하면
Article 9(1)이 개입하여 정당한 이익 근거를 사용할 수 없다.
병원, 종교 시설 등에서 버즈가 분실된 경우가 논리 붕괴 지점이다.

#### 논리 3: 공공장소 합리적 기대의 부재 (Reasonable Expectation)

```
[법리]

공공장소 또는 준공공장소(카페, 지하철, 쇼핑몰)에서는
주변인이 자신의 목소리가 타인에게 우연히 들릴 수 있음을 인식해야 한다.
일시적·비의도적 수음(收音)에 대한 프라이버시 기대 수준은 낮다.

[비교 사례]
  - 스마트 스피커(Alexa, Google Home): 상시 wake word 감지를 위해 주변음 수집
    → 사용자 동의 아래 운영, 제3자 동의 별도 불요
  - ANC 이어폰 자체: 이미 외부 마이크로 주변음 실시간 수집 중
    → 현재도 제3자 동의 없이 수행
  - 유럽 GDPR 감독기관(EDPB)도 "소극적 수음"과 "적극적 개인정보 수집"을 구별하는 경향

[Audi-Find에 유리한 점]
  - 제3자 목소리를 '수집'하는 것이 목적이 아닌 '부수적 포착'
  - 위치 추론 모델이 음성 내용이나 화자를 처리하지 않음
```

**약점**: Mel-Spectrogram이 클라우드 서버로 전송되는 순간
"소극적 수음"의 논리가 유지되기 어렵다.
음향이 서버에 저장·처리되면 "적극적 수집"으로 재분류될 가능성이 높다.
따라서 이 논리는 단독으로 사용하기 어렵고 논리 1·2의 보조 역할에 그친다.

| 조건 | 강화하는 논리 |
|---|---|

```
[현실적 리스크 비교]

전략                   법적 리스크    추가 구현 비용
────────────────────────────────────────────────
VAD 사용               낮음          낮음 (하드웨어 VAD 내장)
논리 1+2+3 결합        중간          낮음 (설계 문서 + 처리방침 정비)
아무 대응 없음         높음          없음

결론:
  논리 1+2+3 결합은 "법적으로 틀렸다"고 단정할 수 없으나,
  GDPR 감독기관 조사 시 방어 입증 부담이 Samsung에게 있다.
  VAD는 추가 지연 0ms + 구현 비용 낮음 + 법적 리스크 대폭 감소.
  → 논리를 보조 방어선으로 갖추되, VAD를 1차 기술적 대응으로 함께 적용하는
    조합이 가장 합리적이다.
```

## 8. 전체 평가 요약

```
현재 ECIES 기반 설계의 기술적 보안 수준:
  ✅ 매우 높음 — 클라우드가 암호문과 키를 동시 보유하지 않음.
  ✅ GDPR Article 32 (기술적·관리적 보안 조치) 요건 충족 수준.
  ✅ Privacy by Design 원칙의 기밀성·무결성 요건 충족.

그러나 기술적 보안 ≠ 법적 준수:

  ❌ 제3자 음성 동의 문제 — VAD 음성 제거 없이는 해결 불가.
  ❌ 상시 수집에 대한 동의 구조 미설계.
  ❌ DPIA 미수행.
  ⚠️ 특수 범주 데이터 별도 동의 체계 미설계.
  ⚠️ 삭제권 이행 API 미설계.

결론:
  기술 설계는 법적 준수의 필요조건이지 충분조건이 아니다.
  암호화 설계를 유지하면서 위 5가지 항목을 보완해야
  GDPR·PIPA·CCPA 기준에서 문제 없이 운용 가능하다.
```
