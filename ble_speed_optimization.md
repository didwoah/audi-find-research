# Audi-Find: BLE 전송 시간 단축 방법 분석

> **기준선**: float16 Mel-Spectrogram, 250 KB = 2,000 Kbits, BLE 실효 처리량 700 Kbps → **2.9초**
> (출처: ble_transmission_analysis.md, 일반 조건 기준)
>
> 목표: 기준선 2.9초를 얼마나 줄일 수 있는가.
> 각 방법은 독립적으로 계산한 뒤, 마지막에 조합 효과를 산출한다.

---

## 전제 및 기호 정의

```
D   = 전송 데이터 크기 (Kbits)
T   = 전송 시간 (s)
BW  = BLE 실효 처리량 (Kbps)

T = D / BW
```

**기준선 수치 재확인**
```
D_base = 250 KB × 8 = 2,000 Kbits
BW_base = 700 Kbps
T_base = 2,000 / 700 = 2.857 s ≈ 2.9 s
```

---

## 방법 1: int8 양자화

### 기술 설명

**양자화(Quantization)** 는 부동소수점(float) 값을 정수(int)로 근사하여 표현 비트 수를 줄이는 기법이다.
float16은 값 하나를 16비트(2바이트)로 표현하고, int8은 8비트(1바이트)로 표현한다.
값의 실제 범위를 미리 파악하여 `scale`과 `zero-point` 두 파라미터를 결정하면, 역변환 시 오차를 최소화할 수 있다.
딥러닝 모델 경량화에서 널리 사용되는 기법으로, 오디오 특성(Mel-Spectrogram)에도 동일하게 적용 가능하다.

### 원리

float16(2B/값) → int8(1B/값)으로 표현 비트 수를 절반으로 줄인다.
Mel-Spectrogram 값의 범위를 사전 측정하여 선형 양자화 파라미터(scale, zero-point)를 결정하면 정밀도 손실이 제한적이다.

### 계산

```
D_int8 = 250 KB × (1/2) × 8 = 1,000 Kbits
BW     = 700 Kbps   (프로토콜 변경 없음)

T_int8 = 1,000 / 700 = 1.429 s
절감   = 2.857 - 1.429 = 1.429 s ≈ 1.4 s 단축
```

### 트레이드오프

- float16 대비 양자화 오차 발생. Mel-Spectrogram 값이 [-80 dB, 0 dB] 범위를 256 단계로 분할하면 분해능 ≈ 0.31 dB → 음향 장면 분류에 허용 가능한 수준으로 예상.
- 클라우드 수신 후 float16으로 역변환하여 AuT 인코더에 입력.

## 방법 2: BLE DLE (Data Length Extension) 활성화

### 기술 설명

**DLE(Data Length Extension)** 는 BLE 4.2에서 도입된 기능으로, 하나의 BLE 패킷에 담을 수 있는 페이로드 크기를 늘리는 프로토콜 협상 기능이다.
BLE는 데이터를 패킷 단위로 전송한다. 각 패킷에는 페이로드 외에 헤더·CRC 등 고정 오버헤드가 붙는다.
DLE 비활성 시 페이로드는 최대 27바이트로 제한되어 같은 데이터를 전송하는 데 훨씬 많은 패킷이 필요하다.
DLE를 활성화하면 페이로드를 최대 251바이트까지 늘려 패킷 수를 줄이고, 헤더 오버헤드 비율을 낮춰 실효 처리량이 증가한다.
별도 하드웨어 변경 없이 연결 협상(LL_LENGTH_REQ/RSP) 메시지만으로 활성화한다.

### 원리

BLE 기본 PDU 페이로드는 **27 bytes**. DLE를 활성화하면 최대 **251 bytes**로 확장된다.
PDU 수가 줄어 패킷당 헤더·CRC 오버헤드 비율이 감소하여 실효 처리량이 증가한다.

### 오버헤드 비율 계산

```
BLE LL 패킷 구조: Preamble(2B) + Access Address(4B) + PDU(가변) + CRC(3B)
PDU 내부: LL Header(2B) + L2CAP Header(4B) + ATT Header(3B) + 페이로드

[DLE 비활성]
  패킷 전체 = 2+4+2+4+3+27+3 = 45 B,  유효 페이로드 = 27 B
  효율 = 27 / 45 = 60.0 %

[DLE 활성]
  패킷 전체 = 2+4+2+4+3+244+3 = 262 B, 유효 페이로드 = 244 B
  효율 = 244 / 262 = 93.1 %

처리량 배율 = 93.1 / 60.0 = 1.55 ×
```

### 계산

```
BW_dle = 700 × 1.55 = 1,085 Kbps
D      = 2,000 Kbits  (데이터 크기 변경 없음)

T_dle  = 2,000 / 1,085 = 1.843 s
절감   = 2.857 - 1.843 = 1.014 s ≈ 1.0 s 단축
```

> **주의**: Galaxy Buds가 이미 DLE를 활성화하고 있을 경우 추가 효과 없음.
> BLE 5.x 스펙상 DLE 지원이 필수이나, 실제 연결 협상에서 활성화 여부는 펌웨어 구현에 따라 다르다.
> 기준선 700 Kbps가 DLE 비활성 상태라면 이 방법의 효과가 유효하다.

## 방법 3: Connection Interval 최소화

### 기술 설명

**Connection Interval(CI)** 은 BLE 연결 기기 두 대가 서로 데이터를 주고받기 위해 라디오를 켜는 주기다.
CI 동안 한 번의 "연결 이벤트(Connection Event)"가 발생하고, 이 이벤트 안에서 여러 패킷을 교환할 수 있다.
CI가 길면 이벤트 횟수가 적어지고, 짧으면 단위 시간당 이벤트 횟수가 늘어난다.
BLE 스펙은 CI를 7.5ms(최소)~4,000ms(최대) 사이에서 1.25ms 단위로 설정할 수 있도록 규정한다.
기본값은 구현마다 다르지만 30~50ms가 일반적이며, 이를 7.5ms로 줄이면 전송 기회가 늘어나 처리량이 향상된다.
단, CI가 짧을수록 라디오가 더 자주 켜지므로 배터리 소모가 증가한다.

### 원리

Connection Interval(CI)은 BLE 기기가 데이터 교환을 위해 슬롯을 열어두는 주기다.
기본값은 일반적으로 30~50ms이며, BLE 스펙 최솟값은 **7.5ms**다.
CI가 짧아질수록 단위 시간당 전송 기회가 늘어나 처리량이 증가한다.

### 계산

대량 데이터 전송 시, CI가 줄어드는 효과는 패킷 스케줄링 밀도 증가에 의한 것이며
패킷당 오버헤드는 변하지 않으므로 개선 폭은 DLE보다 제한적이다.
실측 기반 추정: CI 30ms → 7.5ms 변경 시 처리량 약 **1.3× 향상**.

```
BW_ci = 700 × 1.3 = 910 Kbps
D     = 2,000 Kbits

T_ci  = 2,000 / 910 = 2.198 s
절감  = 2.857 - 2.198 = 0.659 s ≈ 0.7 s 단축
```

### 트레이드오프

- CI를 최소화하면 BLE 라디오 가동 시간이 늘어나 **배터리 소모 증가**.
- 전송 완료 후 CI를 다시 늘리는 적응형 전략으로 완화 가능.

## 방법 4: LZ4 무손실 압축

### 기술 설명

**LZ4** 는 속도를 최우선으로 설계된 무손실 압축 알고리즘이다. Yann Collet이 개발했으며, 압축·복원 속도가 매우 빨라 임베디드 환경에서도 실시간 처리가 가능하다.
LZ4는 데이터 내의 반복 패턴을 찾아 "이전 위치에서 N바이트를 복사하라"는 형태의 참조로 대체하는 방식(LZ77 계열)으로 동작한다.
Mel-Spectrogram은 시간축 방향으로 인접 프레임 간 값 변화가 완만하여(정적인 음향 장면일수록 반복성 높음) 중간 수준의 압축률을 기대할 수 있다.
zlib(Deflate)나 LZMA 등 고압축 알고리즘과 비교해 압축률은 낮지만, 압축 속도가 수백 MB/s 수준으로 Cortex-M55에서 수 ms 이내 처리가 가능하다.

### 원리

int8 Mel-Spectrogram은 시간축 방향으로 인접 프레임 간 값 변화가 완만하다(음향 장면의 통계적 안정성).
LZ4는 경량 무손실 압축 알고리즘으로, Cortex-M55에서도 실시간 처리가 가능하다.
Mel-Spectrogram 특성상 압축률은 제한적이며 **20~30% 감소**를 보수적으로 가정한다.

### 계산

```
D_int8     = 125 KB = 1,000 Kbits      (int8 양자화 이후 기준)
압축률     = 0.75  (25% 감소 가정)
D_lz4      = 1,000 × 0.75 = 750 Kbits

BW         = 700 Kbps  (프로토콜 변경 없음)

T_lz4      = 750 / 700 = 1.071 s
절감 (int8 대비) = 1.429 - 1.071 = 0.358 s ≈ 0.4 s 추가 단축
절감 (기준선 대비) = 2.857 - 1.071 = 1.786 s ≈ 1.8 s 단축
```

### 트레이드오프

- Buds 내부에서 LZ4 압축 연산 추가 (~2–5 ms, 무시 가능).
- 클라우드에서 LZ4 복호화 후 AuT에 입력.
- 압축률은 실제 음향 환경에 따라 달라지며, 순간적으로 소리가 크게 변하는 구간(예: 열차 진입 순간)에서는 압축률이 낮아질 수 있다.

## 방법 5: 선제 전송 (Pre-transmission)

### 기술 설명

**선제 전송** 은 알고리즘·프로토콜 최적화가 아닌 **시스템 아키텍처 전략**이다.
전송 속도 자체를 높이는 것이 아니라, 사용자가 위치 찾기를 요청하기 전에 이미 데이터를 전달해 놓는 방식으로 체감 지연을 0으로 만든다.
스마트폰 메모리나 클라우드에 최신 스냅샷을 상시 캐싱해두고, 요청 시점에 저장된 데이터를 즉시 LLM 추론에 사용한다.
실시간 스트리밍 서비스나 CDN(Content Delivery Network)의 프리페치(prefetch) 전략과 동일한 원리다.

### 원리

"전송 시간을 줄인다"가 아니라 **전송 지연을 0으로 만드는** 아키텍처 전략이다.
BLE 연결이 유지되는 동안 10초 스냅샷을 주기적으로 스마트폰에 미리 전달한다.
위치 찾기 요청 시점에 이미 클라우드에 최신 스냅샷이 존재하므로 전송 대기 없음.

```
사용자 요청 → 전송 대기 없음 → 즉시 LLM 추론 시작
유효 전송 지연 = 0 s
절감 = 2.857 s (기준선 전체 절감)
```

### 트레이드오프

| 이슈 | 내용 |
|---|---|
| 배터리 | 주기적 전송으로 상시 BLE 활성 → 소모 증가 |
| 프라이버시 | 위치 찾기 요청 없이도 오디오 데이터가 클라우드로 전송됨 |
| 스토리지 | 클라우드에 스냅샷 캐싱 필요 |
| 분실 상황 | BLE가 이미 끊긴 상태라면 선제 전송 불가 → 마지막 전송 시점에 의존 |

실용적 절충안: 상시 전송 대신, **RSSI가 임계값 이하로 떨어지거나 연결 불안정 감지 시** 스냅샷 전송을 트리거하는 방식.

## 방법별 단독 효과 요약

| 방법 | 변경 대상 | 기준선 대비 단축 | 전송 시간 |
|---|---|---|---|
| 기준선 | float16, 700 Kbps | — | **2.9 s** |
| ① int8 양자화 | 데이터 크기 250→125 KB | **1.4 s 단축** | 1.4 s |
| ② DLE 활성화 | 처리량 700→1,085 Kbps | **1.0 s 단축** | 1.8 s |
| ③ CI 최소화 (30→7.5ms) | 처리량 700→910 Kbps | **0.7 s 단축** | 2.2 s |
| ④ LZ4 압축 (int8 이후) | 데이터 크기 125→94 KB | **1.8 s 단축** | 1.1 s |
| ⑤ 선제 전송 | 아키텍처 전략 | **2.9 s 절감** | 0 s |

## 가장 현실적인 조합

### 선택 근거

| 방법 | 선택 | 이유 |
|---|---|---|
| ① int8 양자화 | ✅ **채택** | 구현 단순, 정밀도 손실 허용 가능 |
| ② DLE 활성화 | ✅ **채택** | BLE 5.x 표준 기능, 펌웨어 협상만으로 적용 가능 |
| ③ CI 최소화 | ✅ **채택** | 전송 구간에만 적용하는 적응형 전략으로 배터리 영향 최소화 |
| ④ LZ4 압축 | ✅ **채택** | Cortex-M55에서 처리 가능 (~2 ms), 무손실 |
| ⑤ 선제 전송 | ❌ **보류** | 프라이버시·배터리 문제로 설계 검토 필요 |

### 조합 계산

**Step 1: int8 양자화**
```
D₁ = 250 KB × (1/2) = 125 KB = 1,000 Kbits
```

**Step 2: LZ4 압축 적용**
```
D₂ = 1,000 × 0.75 = 750 Kbits   (25% 압축률 가정)
```

**Step 3: DLE 처리량 향상**
```
BW₁ = 700 × 1.55 = 1,085 Kbps
```

**Step 4: CI 최소화 처리량 향상**
```
BW₂ = 1,085 × 1.3 = 1,411 Kbps ≈ 1,400 Kbps
(BLE 2M PHY 실효 최대치 ~1,400 Kbps에 근접하므로 상한으로 고정)
```

**최종 전송 시간**
```
T_opt = D₂ / BW₂ = 750 / 1,400 = 0.536 s ≈ 0.5 s

절감 = T_base - T_opt = 2.857 - 0.536 = 2.321 s ≈ 2.3 s 단축
단축률 = 2.321 / 2.857 × 100 = 81.2 %
```

### 결과 비교

| 항목 | 기준선 | 최적 조합 | 변화 |
|---|---|---|---|
| 데이터 크기 | 250 KB (float16) | 94 KB (int8 + LZ4) | **−62%** |
| BLE 처리량 | 700 Kbps | 1,400 Kbps | **+100%** |
| **전송 시간** | **2.9 s** | **0.5 s** | **−2.3 s (−81%)** |

## 민감도 분석: LZ4 압축률 변동

LZ4 압축률은 실제 음향 환경에 따라 달라진다.

```
[압축률 10% 경우]
D₂ = 1,000 × 0.90 = 900 Kbits
T  = 900 / 1,400 = 0.643 s

[압축률 25% 경우 (기본 가정)]
D₂ = 1,000 × 0.75 = 750 Kbits
T  = 750 / 1,400 = 0.536 s

[압축률 40% 경우]
D₂ = 1,000 × 0.60 = 600 Kbits
T  = 600 / 1,400 = 0.429 s
```

| LZ4 압축률 | 전송 시간 | 기준선 대비 단축 |
|---|---|---|
| 10% | 0.64 s | 2.2 s |
| 25% (가정) | 0.54 s | 2.3 s |
| 40% | 0.43 s | 2.5 s |

압축률 변동에 따른 전송 시간 차이는 0.2초 이내로, DLE·CI 최적화의 효과(처리량 2×)가 지배적이다.
