# Audi-Find: 오디오 수집 트리거 전략 분석

> 주기적 수집 vs 이벤트 기반 수집 vs 하이브리드 방식의 적합성을 논리적으로 분석한다.
> 핵심 목표: BLE 연결이 끊기기 전에 최소 1회 유효한 Mel-Spectrogram을 확보한다.

---

## 1. 문제 정의

```
[수집 트리거의 목적]

버즈가 분실되는 순간에는 대부분 BLE 연결이 이미 끊긴 상태다.
즉, "분실 직전"에 오디오 스냅샷을 확보해야 나중에 Find 요청 시 활용할 수 있다.

타이밍 조건:
  ① 스냅샷 수집 완료
  ② Mel-Spectrogram 변환 (~10 ms)
  ③ AES-256-GCM 암호화 (~5 ms)
  ④ BLE 전송 (버즈 → 스마트폰, 0.5~2.9 초)
  ⑤ 스마트폰 → 클라우드 업로드

→ ①~④가 BLE 연결이 끊기기 전에 완료되어야 함.
   BLE 연결이 끊긴 후에는 Finder 기기가 클라우드에 업로드한 데이터만 사용 가능.
```

---

## 2. 핵심 파라미터 정리

### 2.1 BLE 연결 유효 반경

```
실내 BLE 유효 연결 반경 (장애물 있는 환경):
  BLE 5.3 / 5.4 (2M PHY): 이론 최대 200m, 실내 실효 10~15m
  → 보수적 기준: 10m (콘크리트 벽·사람·가구 간섭 고려)

RSSI 기반 연결 단계:
  -50 dBm 이하:  안정 연결 (밀착 근거리)
  -70 dBm:       신호 약화 시작 (약 6~8m 추정)
  -80 dBm:       연결 불안정 경고 (약 9~10m)
  -85 dBm 이하:  하드 연결 해제(hard disconnect) 발생 구간
```

### 2.2 사람의 이동 속도

```
시나리오별 속도:

  느린 걷기:   1.0 m/s   (쇼핑몰, 붐비는 지하철)
  보통 걷기:   1.4 m/s   (성인 평균 보행 속도)
  빠른 걷기:   1.8 m/s   (바쁜 출퇴근)
  달리기:      3.0 m/s   (조깅 수준)
```

### 2.3 BLE 전송 소요 시간

```
ble_speed_optimization.md 기준:

  최악 (baseline, float16):  2.9 초
  최적 (int8 + LZ4 + DLE):   0.5 초

→ 분석에서 두 경우를 모두 고려함.
```

## 3. 전략 A: 주기적 수집 (Periodic Collection)

### 3.1 수집 주기 결정 논리

**전제**: 버즈는 고정되어 있고, 스마트폰(소유자)이 멀어지는 상황.

```
[BLE 단절까지 남은 시간 계산]

스마트폰이 버즈로부터 직선 방향으로 걷는 경우:
  유효 반경: R = 10 m
  소요 시간 T_walk = R / v

  v = 1.0 m/s → T_walk = 10.0 초
  v = 1.4 m/s → T_walk =  7.1 초  (가장 현실적인 기준)
  v = 1.8 m/s → T_walk =  5.6 초
  v = 3.0 m/s → T_walk =  3.3 초
```

주기 수집이 성공하려면 주기 T_period가 다음 조건을 만족해야 한다.

```
[성공 조건]

마지막 수집 시점이 BLE 단절 T_walk 이내에 있어야 함.
수집 후 전송까지 T_tx 소요.

필요 조건:
  T_period + T_tx < T_walk

→ T_period < T_walk - T_tx

경우별 계산:

  최악 전송 (T_tx = 2.9s):
    v = 1.4 m/s: T_period < 7.1 - 2.9 = 4.2 초 → 최대 4초 주기
    v = 1.8 m/s: T_period < 5.6 - 2.9 = 2.7 초 → 최대 2초 주기
    v = 3.0 m/s: T_period < 3.3 - 2.9 = 0.4 초 → 사실상 불가

  최적 전송 (T_tx = 0.5s):
    v = 1.4 m/s: T_period < 7.1 - 0.5 = 6.6 초 → 최대 6초 주기
    v = 1.8 m/s: T_period < 5.6 - 0.5 = 5.1 초 → 최대 5초 주기
    v = 3.0 m/s: T_period < 3.3 - 0.5 = 2.8 초 → 최대 2초 주기
```

### 3.2 주기별 배터리 영향 추정

```
수집 1회 에너지 소비:
  Mel-Spectrogram 변환:  Cortex-M55 @ 200 MHz, ~10 ms
    → 활성 전류 ~5 mA (12nm, Helium 활성화 기준)
    → 소비 전하: 5 mA × 0.01 s = 0.05 mC

  AES-256-GCM 암호화: ~5 ms, ~5 mA → 0.025 mC

  BLE 전송 (최적 0.5s): 전송 전류 ~15 mA
    → 소비 전하: 15 mA × 0.5 s = 7.5 mC

  수집 1회 총 전하: ~7.6 mC

버즈2 Pro 배터리: 61 mAh = 219,600 mC

주기별 배터리 소모 (하루 8시간 착용 기준):

  T_period = 6s:    8 × 3600 / 6 = 4,800 회/일
    소비: 4,800 × 7.6 mC = 36,480 mC = 36.5 mAh
    → 배터리의 59.8% 소모 ← 재생 시간 절반 수준으로 감소, 심각한 영향

  T_period = 30s:   960 회/일
    소비: 7,296 mC = 7.3 mAh → 배터리의 12.0% 추가 소모 (수용 가능)

  T_period = 60s:   480 회/일
    소비: 3,648 mC = 3.6 mAh → 배터리의 5.9% 추가 소모 (양호)

  T_period = 300s:  96 회/일
    소비: 730 mC = 0.73 mAh → 배터리의 1.2% 추가 소모 (무시 가능)
```

### 3.3 주기 수집의 커버리지 vs 배터리 트레이드오프

```
T_period   커버 가능 최대 속도 (T_tx=0.5s 기준)   배터리 추가 소모
──────────────────────────────────────────────────────────────────
  2 초      v < (10 - 0.5) / 2 = 4.75 m/s → 달리기 포함    대폭 감소
  6 초      v < (10 - 0.5) / 6 = 1.58 m/s → 보통 걷기 가능  심각 (59%)
 30 초      v < (10 - 0.5) /30 = 0.32 m/s → 매우 느린 이동만 포함  12%
 60 초      v < 0.16 m/s → 거의 정지 상태만 포함             6%
```

**결론**: 주기 수집만으로 보통 걷기(1.4 m/s)를 커버하려면 6초 이하 주기가 필요하나,
배터리 소모가 59%에 달해 단독 전략으로 채택하기 어렵다.

## 4. 전략 B: 이벤트 기반 수집 (Event-Triggered Collection)

### 4.1 활용 가능한 센서 (BES2700YP 기준)

```
BES2700YP + 탑재 센서 (buds_hardware.md 참조):

  ① 하드웨어 VAD        — 음성/비음성 판별, 착용 여부 간접 감지
  ② RSSI 모니터링       — BLE 스택에서 실시간 신호 강도 측정
  ③ LSM6DSV16BX IMU    — 가속도계(±16g) + 자이로(±2000 dps), 6축
  ④ 인이어(In-Ear) 감지 — 착용 감지 회로 (대부분 Buds Pro 계열 탑재)
```

### 4.2 트리거 이벤트 설계

#### 트리거 1: RSSI 임계값 하강

```
[동작 원리]

BLE 스택에서 RSSI를 주기적으로 측정 (BLE 5.x: 연결 이벤트마다 갱신, ~30ms 간격).
RSSI가 임계값 아래로 내려가면 즉시 오디오 수집 → 전송 시작.

임계값 설정:
  경고 임계값: -70 dBm (약 6~8m, 단절 전 1.5~3초 여유)
  긴급 임계값: -78 dBm (약 9m, 단절 전 0.5~1.5초 여유)

[여유 시간 계산]

경고(-70 dBm) → 단절(-85 dBm) 거리 차: 약 2m
소요 시간: 2 m / 1.4 m/s = 1.4 초

긴급(-78 dBm) → 단절(-85 dBm) 거리 차: 약 1m
소요 시간: 1 m / 1.4 m/s = 0.7 초

→ 경고(-70 dBm) 기준: 전송에 최대 1.4초 확보
  최적 전송 (0.5s): 성공 가능 (여유 0.9s)
  기본 전송 (2.9s): 실패 가능성 높음

→ 긴급(-78 dBm) 기준: 최적 전송도 위험
```

```
[이중 임계값 전략]

RSSI = -70 dBm 돌파:
  경고 수집 시작. Mel-Spectrogram 10초치 준비 → 전송 시작.
  전송 완료 전에 단절되면 버즈 내부 버퍼에 암호화된 스냅샷 보관.
  → Finder 기기가 나중에 GATT로 수신 가능.

RSSI = -78 dBm 돌파:
  긴급 상태. 현재 10초 버퍼를 즉시 프리즈(freeze)하고
  최소 크기(int8 + LZ4 압축)로 재암호화 후 긴급 전송.
```

#### 트리거 2: 착용 해제 (In-Ear 감지)

```
[동작 원리]

인이어 센서(광학식 PPG 또는 정전용량 방식)가 착용 해제를 감지.
→ 사용자가 귀에서 버즈를 빼는 행위 = 잠시 후 케이스에 넣거나 어딘가에 놓을 가능성.

트리거 조건:
  착용 해제 AND (RSSI < -55 dBm)  ← 이미 중거리에 있는 경우만
  → "집에서 귀에서 빼고 케이스에 넣는" 상황 등 false positive 방지

대응:
  착용 해제 감지 즉시 수집 시작 + 전송.
  → 버즈를 다시 착용하면 수집 중단 (불필요 전송 방지).
```

#### 트리거 3: 이상 가속도 감지 (낙하·충격)

```
[동작 원리]

낙하 패턴 인식 (LSM6DSV16BX 가속도계):
  자유낙하 구간: 가속도 < 0.3g (중력 제거) 지속 50ms 이상
  충격: 이후 |a| > 3g 스파이크 (지면 충돌)

→ "버즈가 귀에서 떨어졌음"을 감지

대응:
  낙하 감지 후 500ms 이내 오디오 수집 + 즉시 전송.
  낙하 직전 10초 순환 버퍼가 분실 장소 오디오를 포함.
```

#### 트리거 4: IMU 정지 + 스마트폰 이동 감지

```
[동작 원리]

버즈 IMU → 가속도 변화 없음 = 버즈가 정지 상태 (사물 위에 놓임)
스마트폰 IMU (소유자 폰) → 보행 패턴 감지 = 소유자가 이동 중
→ "버즈는 놓여 있고 소유자는 걷고 있음" = 분실 전조

트리거 조건:
  버즈 정지 (|a| < 0.05g, 5초 이상) AND RSSI 하강 추세
  → 수집 + 전송 시작
```

### 4.3 이벤트 기반 수집의 배터리 영향

트리거 횟수는 실사용 로그 없이는 결정할 수 없다. 이 분석에서는 트리거 횟수를 변수 N으로 두고, 현실적인 범위를 추정한다.

```
[1회 트리거 소비 에너지 — 계산 근거 있는 고정값]

  Mel-Spectrogram 변환: 10 ms × 5 mA = 0.05 mC
  AES-256-GCM 암호화:   5 ms × 5 mA = 0.025 mC
  BLE 전송 (최적 0.5s): 0.5 s × 15 mA = 7.5 mC
  합계: 약 7.6 mC / 회

버즈2 Pro 배터리: 61 mAh = 219,600 mC
```

```
[트리거 횟수 N에 따른 배터리 소모]

  N = 10 회/일:  76 mC  = 0.08 mAh = 배터리의 0.1%
  N = 30 회/일:  228 mC = 0.23 mAh = 배터리의 0.4%
  N = 100 회/일: 760 mC = 0.77 mAh = 배터리의 1.3%
  N = 300 회/일: 2,280 mC = 2.3 mAh = 배터리의 3.8%
```

```
[트리거 횟수 N의 상한 추정]

트리거별 발생 가능 최대 횟수 (하루 8시간 착용):

  RSSI 트리거 (-70 dBm 하강):
    BLE 연결 이벤트마다 RSSI 갱신 (~30ms 간격).
    -70 dBm 임계값을 여러 번 넘나들 수 있으므로
    히스테리시스(hysteresis) 없이는 단일 이동에서 수십 회 발생 가능.
    → 히스테리시스 적용 (재발동 억제 30초) 시: 최대 8시간 / 30초 = 960회
    → 현실적으로 RSSI가 임계값 근처에서 진동하는 시간이 제한적이므로
       히스테리시스 없이도 수십 회 수준 예상.

  착용 해제 트리거:
    하루 착용/탈착 횟수 상한은 행동 패턴에 따라 다르지만
    식사·회의·통화 등을 고려하면 최대 20회 수준.

  낙하 감지:
    정상 사용 시 0~2회 (가방 안 충격, 운동 중).

  IMU 정지 트리거 (조건: 버즈 정지 5초 AND RSSI 하강 추세):
    RSSI 하강 추세 조건이 필터 역할을 함.
    RSSI 안정 구간(집, 사무실 고정)에서는 발동 안 됨.
    최대 이동·이탈 상황에서만 발동 → 현실적으로 RSSI 트리거와 겹침.

→ 히스테리시스·중복 방지 로직 적용 시 하루 N = 10~50회 수준이 현실적 범위.
   보수적 상한: N = 100회 (배터리 소모 1.3%) 로 설계 여유 확보.
```

**결론**: 이벤트 기반 수집의 배터리 소모는 히스테리시스·중복 방지 구현 품질에 크게 의존한다. 1회당 7.6 mC는 고정값이므로, N을 제어하는 것이 핵심이다. N ≤ 100회/일 설계를 목표로 하면 배터리 추가 소모 **1.3% 이내**를 보장할 수 있다.

## 5. 전략 C: 하이브리드 (주기 수집 + 이벤트 트리거)

### 5.1 주기 후보 비교: 30초 vs 1분

주기 수집의 역할은 이벤트 트리거가 발동하지 못한 상황(전파 음영 급단절 등)의 **최후 백업**이다. 이 관점에서 두 주기를 비교한다.

```
[주기별 배터리 소모 — 상시 연결(RSSI 안전권) 가정]

수집 1회 소비: 7.6 mC (3절 계산 기준)
하루 8시간 착용 기준:

  30초 주기: 8 × 3600 / 30 = 960 회/일
    소비: 960 × 7.6 = 7,296 mC = 7.3 mAh → 배터리 61mAh의 12.0%

  1분 주기:  8 × 3600 / 60 = 480 회/일
    소비: 480 × 7.6 = 3,648 mC = 3.6 mAh → 배터리 61mAh의  5.9%

  차이: 3.7 mAh (재생 시간으로 환산 시 약 18분 차이)
```

```
[주기별 스냅샷 신선도 — 분실 장소와의 거리 오차]

"마지막 수집 이후 T_period 동안 이동한 거리"가 위치 오차가 된다.

  30초 주기, 보통 걷기 1.4 m/s:  1.4 × 30 = 42 m 오차
  1분 주기,  보통 걷기 1.4 m/s:  1.4 × 60 = 84 m 오차

GPS 후보 반경 기준 (gps_accuracy.md):
  Offline Finding 최대 오차 = 100 m

  42 m 오차 → GPS 반경(100m) 내에 충분히 포함. 음향 지문 유효.
  84 m 오차 → GPS 반경 경계 근처. 유효하나 여유 감소.

→ 두 주기 모두 GPS 반경 100m 이내 오차로 위치 판별에 활용 가능.
   단, 30초 주기가 오차 범위 면에서 더 안전하다.
```

```
[주기별 프라이버시 영향]

8시간 착용 기준 수집 횟수:
  30초 주기: 960 회/일
  1분 주기:  480 회/일

privacy_compliance.md 관점:
  상시 수집은 "최소 수집 원칙" 위반 소지 → 횟수가 적을수록 유리.
  두 주기 모두 동일한 법적 구조(명시적 동의 + 처리방침 기재) 필요.
  횟수 자체보다 동의 여부와 처리 목적의 단일성이 핵심.
```

### 5.2 30초/1분 주기 × 이벤트 트리거 하이브리드 설계

```
[동작 구조]

  ┌─────────────────────────────────────────────┐
  │  RSSI 구간        주기 백업    이벤트 트리거  │
  │                                             │
  │  > -55 dBm  →   1분 주기  +  P2·P3·P4 감시 │  ← 안전 구간
  │  -55~-70 dBm →  30초 주기 +  P2·P3·P4 감시 │  ← 경계 구간
  │  < -70 dBm  →   P1 즉시 발동 (주기 대체)    │  ← 위험 구간
  └─────────────────────────────────────────────┘

이벤트 트리거 우선순위:
  P1 (즉시): RSSI < -70 dBm 돌파       ← 주기를 대체
  P2 (즉시): 낙하 감지 (자유낙하 + 3g 충격)
  P3 (즉시): 착용 해제 AND RSSI < -55 dBm
  P4 (즉시): IMU 정지 5초 AND RSSI 하강 추세

중복 방지:
  이벤트 또는 주기 수집 성공 후 60초 이내 재수집 억제.
  → 주기와 이벤트가 같은 시점에 겹쳐 두 번 전송하는 낭비 방지.
```

### 5.3 하이브리드의 배터리 총 소모 추정

```
[케이스별 배터리 소모]

A. 안전 구간(RSSI > -55 dBm) 위주 하루 (집, 사무실 고정):
   주기: 1분 → 5.9%
   이벤트: 거의 없음 → ~0%
   총: ~5.9%

B. 이동 많은 하루 (출퇴근 + 외출 포함):
   안전 구간 4시간: 240회 × 7.6 mC = 1,824 mC
   경계 구간 2시간 (30초 주기): 240회 × 7.6 mC = 1,824 mC
   위험 구간 이벤트: 히스테리시스 적용 시 최대 20회 × 7.6 mC = 152 mC
   합계: 3,800 mC = 3.8 mAh = 배터리의 6.2%

C. 고밀도 이동 (하루 종일 외출, 지하철 포함):
   경계 구간 6시간 (30초 주기): 720회 × 7.6 mC = 5,472 mC
   이벤트 최대 100회: 760 mC
   합계: 6,232 mC = 6.3 mAh = 배터리의 10.3%

→ 일반 사용 기준 배터리 추가 소모: 약 6~7%
   재생 시간(ANC ON 기준 5시간) 기준 영향: 약 18~21분 감소
```

### 5.4 30초 주기 선택의 논리적 근거

#### 근거 1: GPS 반경 100m 내 유효 음향 지문 확보

주기 수집의 핵심 목표는 **Find 요청 시 GPS 후보 반경(100m) 안의 음향 환경을 포착하는 것**이다. 스냅샷이 이 범위를 벗어나면 분실 장소와 음향적으로 다른 환경의 데이터가 전달되어 위치 판별 정확도가 하락한다.

```
[주기 T일 때 최대 위치 오차 계산]

최악의 경우: 수집 직후 BLE 단절
→ 단절 시점까지 T초 동안 소유자가 이동

  이동 거리 D = v × T

속도별 D (T = 30s):
  느린 걷기  1.0 m/s → D = 30 m
  보통 걷기  1.4 m/s → D = 42 m  ← 기준 시나리오
  빠른 걷기  1.8 m/s → D = 54 m
  달리기     3.0 m/s → D = 90 m  ← GPS 반경 경계

속도별 D (T = 60s):
  느린 걷기  1.0 m/s → D = 60 m
  보통 걷기  1.4 m/s → D = 84 m
  빠른 걷기  1.8 m/s → D = 108 m ← GPS 반경(100m) 초과
  달리기     3.0 m/s → D = 180 m ← 완전히 벗어남
```

```
[GPS 반경 활용률 비교]

GPS 반경 = 100 m (gps_accuracy.md)
"활용률" = D / 100m × 100% → 낮을수록 분실 장소에 가까운 오디오

                 30초 주기   1분 주기
  보통 걷기:      42%        84%    ← 1분은 반경 거의 소진
  빠른 걷기:      54%       108%    ← 1분은 반경 초과
  달리기:         90%       180%    ← 30초도 경계

→ 1분 주기는 빠른 걷기(1.8 m/s) 이상에서 GPS 반경을 벗어남.
   30초 주기는 달리기(3.0 m/s)를 제외하면 반경 내에 유지됨.
```

**30초의 논리**: 성인 이동 속도의 대다수(느린 걷기~빠른 걷기, 약 80% 이상)를 GPS 반경 내에 포함시키는 최장 주기가 30초다. 이보다 길면 빠른 걷기 시나리오에서 음향 지문이 반경 밖에서 수집될 위험이 생긴다.

---

#### 근거 2: 이벤트 트리거 미발동 시나리오 대응

이벤트 트리거(RSSI, 낙하, 착용 해제)가 발동하지 못하는 경우가 존재한다.

```
[이벤트 트리거 실패 시나리오]

  ① 전파 음영 급단절 (지하철 터널 진입, 지하주차장)
       RSSI가 점진적으로 하강하지 않고 갑자기 -90 dBm 이하로 떨어짐.
       → RSSI 경고 트리거(-70 dBm) 발동 불가.
       → 주기 수집 데이터가 유일한 스냅샷.

  ② 착용 중 단절 (소유자가 이어폰 착용한 채 범위 벗어남)
       착용 해제 트리거 발동 없음.
       → RSSI 트리거만 의존.
       → RSSI 급단절 시 주기 수집 데이터에 의존.

이 두 시나리오에서 주기 수집이 없다면 스냅샷 = 없음.
```

```
[시나리오별 급단절 빈도 추정]

지하철 1회 탑승 기준:
  터널 진입 시 급단절: 탑승 중 수 회
  지하역 → 지상: 신호 복원

지하주차장 진입:
  1회 주차 시 1회 급단절

→ 급단절은 이벤트 트리거 실패가 가장 빈번한 시나리오.
   지하철 통근자에게는 하루 수 회 발생 가능.
```

30초 주기는 이 시나리오에서 최대 42m 오차(보통 걷기 기준)로 대응한다. 지하철 터널 진입 직전의 환경음(플랫폼 음향)이 포착되어 "지하철역"을 식별하는 데 충분히 유효하다.

---

#### 근거 3: 20초·15초가 아닌 30초인 이유 (하한 결정)

더 짧은 주기도 정확도 면에서 유리하지만 배터리 비용이 급증한다.

```
[주기별 배터리 소모 — 안전 구간 8시간 착용]

  15초 주기: 1,920 회 × 7.6 mC = 14,592 mC = 14.6 mAh = 배터리의 23.9%
  20초 주기: 1,440 회 × 7.6 mC = 10,944 mC = 11.0 mAh = 배터리의 18.0%
  30초 주기:   960 회 × 7.6 mC =  7,296 mC =  7.3 mAh = 배터리의 12.0%
  60초 주기:   480 회 × 7.6 mC =  3,648 mC =  3.6 mAh = 배터리의  5.9%

재생 시간(ANC ON 5시간) 환산 손실:
  15초 주기: 71분 감소 (5h → 3h 49m)  ← 체감 심각
  20초 주기: 54분 감소 (5h → 4h 6m)   ← 체감 심각
  30초 주기: 36분 감소 (5h → 4h 24m)  ← 허용 가능
  60초 주기: 18분 감소 (5h → 4h 42m)  ← 거의 무감
```

```
[위치 오차 대비 배터리 비용 효율]

주기를 절반으로 줄이면 배터리 소모가 2배가 되고,
위치 오차는 절반으로 줄지만 GPS 반경(100m) 대비 이미 충분한 여유가 있는지 확인.

  60초 → 30초: 오차 84m → 42m (42m 개선), 배터리 +6.1% 추가
  30초 → 20초: 오차 42m → 28m (14m 개선), 배터리 +6.0% 추가
  20초 → 15초: 오차 28m → 21m ( 7m 개선), 배터리 +5.9% 추가

60초 → 30초 구간에서 개선 효과(42m)가 가장 크고,
30초 이하로 줄일수록 개선 폭이 급감하는 반면 배터리 비용은 동일하게 증가.
→ 30초가 배터리 대비 위치 오차 개선 효율이 가장 높은 지점.
```

### 5.6 30초 vs 1분 최종 비교

| 항목 | 30초 주기 + 이벤트 | 1분 주기 + 이벤트 |
|---|---|---|
| **전파 음영 급단절 시 최대 위치 오차 (1.4 m/s)** | 42 m | 84 m |
| **빠른 걷기(1.8 m/s) 시 GPS 반경 초과 여부** | ❌ 초과 안 함 (54m) | ✅ 초과 (108m) |
| **달리기(3.0 m/s) 시 GPS 반경 초과 여부** | ✅ 경계 (90m) | ✅ 초과 (180m) |
| **일반 하루 배터리 추가 소모** | 10~13% | 5~7% |
| **재생 시간 감소 (ANC ON 5h 기준)** | 30~40분 | 15~21분 |
| **프라이버시 노출 횟수** | 960 회/일 | 480 회/일 |
| **배터리 대비 오차 개선 효율** | ✅ 가장 높음 | (기준) |

```
결론:

  30초 주기는 빠른 걷기(1.8 m/s)까지 GPS 반경(100m) 안에 스냅샷을 유지하는
  최장 주기이자, 배터리 대비 위치 오차 개선 효율이 가장 높은 지점이다.

  1분 주기는 배터리·프라이버시 우선 사용자에게 합리적이나,
  빠른 걷기 이상 속도에서 GPS 반경을 벗어나 음향 지문의 유효성이 떨어질 수 있다.

실용적 권고:
  기본값    → 1분 주기 (배터리·프라이버시 우선, 15~21분 손실)
  고급 설정 → 30초 주기 (정확도 우선, 32~40분 손실)
             지하철 통근자·외근직 등 급단절 빈도 높은 사용자에게 권장
```

## 7. 최종 권고

```
권고: 1분 주기 백업 + 이벤트 트리거 하이브리드 (기본값)
      30초 주기 백업 + 이벤트 트리거 하이브리드 (고급 설정)

[트리거 우선순위]

  P1 (즉시): RSSI < -70 dBm             ← 단절 1.4초 전 예고, 주기 수집 대체
  P2 (즉시): 낙하 감지 (자유낙하 + 3g)   ← 물리적 분실 이벤트
  P3 (즉시): 착용 해제 AND RSSI < -55 dBm ← 소유자 이탈 조짐
  P4 (즉시): IMU 정지 5초 AND RSSI 하강   ← 버즈 놓임 + 소유자 이동
  P5 (주기): RSSI > -55 dBm → 1분 주기
             RSSI -55~-70 dBm → 30초 주기
             RSSI < -70 dBm  → P1 발동 (주기 대체)

[핵심 설계 근거]

  이벤트 트리거(P1~P4):
    RSSI -70 dBm에서 단절까지 약 1.4초 여유.
    최적 전송(0.5s)으로 여유 내 완료 가능.
    낙하·착용 해제 트리거는 물리적 분실 상황 직접 대응.

  1분 주기 백업:
    이벤트 트리거가 발동하지 못한 전파 음영 급단절 대응.
    최대 위치 오차 84m → GPS 반경(100m) 이내 포함.
    배터리 추가 소모 ~6%, 재생 시간 약 18분 감소.

  30초 주기 옵션:
    위치 오차 42m로 정확도 향상.
    배터리 추가 소모 ~10%, 재생 시간 약 30분 감소.
    지하철·지하주차장 등 급단절 빈도 높은 사용자에게 권장.
```
